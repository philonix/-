Isolation Forest 精炼分析摘要

1. 数据说明

本次分析基于以下数据集完成：

1. 训练集：run_normal_001 ~ run_normal_003
2. 验证集：run_001、run_003、run_005
3. 测试集：run_002、run_004、run_006
4. 挑战集：run_007、run_008、run_009、run_010

当前分析对象聚焦于 carts 服务。

2. 最终采用的模型配置

当前最优配置来自特征组与参数联合对比结果：

1. n_estimators = 300
2. contamination = 0.08
3. selected_feature_set = rate_plus_rolling

rate_plus_rolling 特征组由以下特征构成：

1. cpu_rate
2. rolling_mean
3. rolling_std
4. abs_rate_change

说明：

1. 在当前数据上，单纯增加更多统计特征没有稳定带来提升
2. 引入 memory 特征后，整体效果未优于 rate_plus_rolling
3. 因此当前阶段保留 CPU 侧的核心滚动统计特征作为主方案

3. 当前定量结果

总体阶段结果如下：

1. validation：
   - avg_precision = 0.0753
   - avg_recall = 0.4896
   - avg_f1 = 0.1291
2. test：
   - avg_precision = 0.0849
   - avg_recall = 0.2665
   - avg_f1 = 0.1272
3. challenge：
   - avg_precision = 0.0941
   - avg_recall = 0.2567
   - avg_f1 = 0.1343

说明：

1. 当前模型在 challenge 集上的平均 F1 略高于 test 集
2. precision 与 recall 都不高，说明模型具备一定定位能力，但边界仍不准确

4. 分轮次表现

测试集逐轮结果：

1. run_002：F1 = 0.0833
2. run_004：F1 = 0.1124
3. run_006：F1 = 0.1860

挑战集逐轮结果：

1. run_007：F1 = 0.2373
2. run_008：F1 = 0.1474
3. run_009：F1 = 0.1000
4. run_010：F1 = 0.0526

说明：

1. run_007 的识别效果相对最好
2. run_010 的识别效果最差，说明复杂、随机性更强的复合故障更难检测
3. 不同轮次波动较大，模型对不同扰动模式的适应性仍不稳定

5. 当前问题定位

从趋势图与异常点分布观察，当前最主要的问题不是完全检不出异常，而是：

1. 模型通常能在故障段附近识别出异常
2. 但在故障段前后存在较多误报
3. 尤其是故障前后的过渡区、恢复区，很多正常点被标为异常

这说明：

1. 模型已经具备粗粒度定位能力
2. 但点级判断的异常边界不够精确
3. 当前主要瓶颈是误报过多，而不是漏报为主

6. 对当前结果的合理结论

可以得出的结论是：

1. Isolation Forest 在当前 Sock Shop 故障注入数据上能够对异常区段产生响应
2. 模型能在多数实验轮次中定位到故障附近区域
3. 但其点级预测存在较多误报，导致 precision 偏低
4. 说明当前方案更适合作为“故障区域粗定位工具”，尚未达到高精度告警器水平

7. 下一步优化方向

建议优先做以下两类优化，而不是继续无约束堆特征：

1. 异常后处理
   - 仅保留连续 2~3 个点的异常片段
   - 去掉孤立异常点
   - 对异常分数做平滑
2. 评估方式升级
   - 从点级评估补充到事件级命中评估
   - 允许故障前后一定的时间容忍带
   - 单独识别恢复期，不将其全部视作严格误报

这样做的目标不是改原始数据，而是在现有结果基础上减少故障邻域的误报。

9. 当前精炼结论

一句话概括当前成果：

模型已经能识别“问题大致发生在哪”，但会把故障段附近过多正常点一起标成异常；因此当前最需要解决的是误报控制和异常边界收缩，而不是继续扩大原始数据规模。
