消融与稳健性实验整理说明

1. 本阶段定位

由于模型优化已经暂时停止，本文件不再设计新的大规模搜索，而是整理已有结果中最有价值的消融和稳健性结论。

这里的“后续实验”主要包括：

1. 特征组消融
2. 后处理消融
3. 测试集与挑战集稳健性分析
4. 困难轮次误差分析

2. 特征组消融结论

V1 阶段：

1. rate_plus_rolling 的 avg_f1 = 0.1361，高于 full_features 的 0.1291
2. 说明在原始基线阶段，轻量 CPU 侧滚动统计特征已经足够形成基本检测能力

V3 阶段：

1. full_features 的 avg_f1 = 0.1184，为最优
2. stat_enhanced 的 avg_f1 = 0.1066
3. cpu_memory 的 avg_f1 = 0.0496
4. rate_only 的 avg_f1 = 0.0743
5. rate_plus_rolling 的 avg_f1 = 0.0519

V4 阶段：

1. full_features 的 avg_f1 = 0.1158，为最优
2. stat_enhanced 的 avg_f1 = 0.0854
3. rate_plus_rolling 的 avg_f1 = 0.0857
4. cpu_memory 的 avg_f1 = 0.0816
5. rate_only 的 avg_f1 = 0.0620

从这些结果可以得出：

1. 在 V1 基线阶段，轻量特征组更利于形成粗粒度定位
2. 进入 V3/V4 后，full_features 才真正体现出优势
3. 仅靠 cpu_rate 或轻量 rolling 特征，不足以稳定支撑误报控制和 fault-aware 选型
4. memory 相关特征单独使用并不强，但与 full_features 联合时是有价值的

因此，当前最合理的特征消融结论是：

1. 主版本仍保留 full_features
2. rate_plus_rolling 可作为早期基线对照
3. rate_only 不适合作为最终版本特征组

3. 后处理消融结论

V2 的固定后处理：

1. streak_2
2. 优点是事件覆盖率较高
3. 缺点是仍容易保留较宽的故障邻域

V3 的关键变化：

1. 从 streak_2 收紧到 streak_3
2. 同时对 smooth_window 和 vote_threshold 做联合比较
3. 最终主版本仍选择不额外平滑的 streak_3

V3 排名前列结果显示：

1. 高覆盖配置通常对应 contamination=0.12
2. 它们虽然 event_hit_rate 可以达到 1.0
3. 但 anomaly_ratio 经常超过 0.55，甚至到 0.62 以上

V4 排名前列结果显示：

1. 最高分 fault-aware 候选往往也集中在 contamination=0.12 或 0.10
2. 这些配置虽然 kill_hit_rate 和 pause_hit_rate 都能接近 1.0
3. 但 anomaly_ratio 往往达到 0.48~0.72，明显过高

这说明：

1. 平滑和更宽松的连续阈值确实能带来更高覆盖
2. 但它们会连带扩大误报区域
3. 当前任务的主要矛盾仍然是“如何让异常边界更窄”
4. 所以 `streak_3 + 尽量少平滑` 依然是较合理的主版本后处理选择

4. 稳健性分析

从 test 和 challenge 结果看，模型表现存在明显的阶段差异。

V3：

1. test avg_f1 = 0.1838
2. challenge avg_f1 = 0.1646
3. precision 在 test 上明显更高
4. 说明 V3 在更复杂挑战场景下仍有一定退化，但退化幅度可接受

V4：

1. test avg_f1 = 0.1861
2. challenge avg_f1 = 0.1493
3. challenge 下的下降幅度比 V3 更明显
4. 说明 V4 对复杂场景的泛化还不够稳定

因此，稳健性层面的主结论是：

1. V3 的跨阶段稳定性优于 V4
2. V4 更容易受挑战集复杂轮次影响
3. 如果后续报告强调泛化与稳定，主版本必须选 V3

5. 困难轮次分析

从多个版本结果看，较难的轮次高度集中在 challenge 集的后半部分。

重点困难轮次：

1. run_009
2. run_010

现象总结：

1. run_010 在 V1/V2/V3/V4 中都偏难
2. 说明问题不只是某一版参数设置错误
3. 更可能是复合故障、恢复段重叠和时序波动共同导致

V3 与 V4 对比：

1. V4 对 run_009 的覆盖有所改善
2. 但对 run_010 仍然没有根本解决
3. 同时 V4 会把更多邻近正常点一起卷入异常

因此在最终报告里，run_010 很适合作为“当前模型极限与不足”的典型案例。

6. 建议在最终报告里如何写这些后续实验

建议把后续实验整理成三条线：

1. 特征消融线：说明 full_features 为什么最终胜出
2. 后处理消融线：说明为什么主版本不选高覆盖宽松策略
3. 稳健性线：说明为什么主版本必须优先考虑 challenge 集表现

7. 一句话总结

现有消融和稳健性结果已经足够支撑一个完整结论：当前最佳主版本不是覆盖最宽的版本，也不是针对单一故障类型命中率最高的版本，而是能够在 test 和 challenge 上同时保持误报可控的 V3。
