冻结模型后的版本对比总结

1. 本阶段目标

本阶段不再继续新开 V5 或继续调参，而是把已有版本结果整理成可用于后续实验报告的统一结论。

当前参与对比的版本为：

1. V1：原始基线版本
2. V2：加入连续异常后处理和事件级评估的版本
3. V3：以误报控制为主的平衡版本
4. V4：以 pause 命中率为重点的 fault-aware 补充版本

2. 版本演化逻辑

V1 -> V2：

1. 从纯点级评估推进到事件级评估
2. 加入最基本的连续异常过滤
3. 目标是让模型更稳定地碰到真实故障事件

V2 -> V3：

1. 重新评估 contamination 是否过大
2. 收紧后处理，从 streak_2 过渡到 streak_3
3. 目标是压缩故障段前后的误报

V3 -> V4：

1. 不再只看 overall 指标
2. 显式加入 kill_hit_rate 和 pause_hit_rate
3. 目标是避免 pause 在主配置筛选时被系统性牺牲

4. 直接结论

第一点：V2 相比 V1 已经证明分析层优化有效

1. test F1 明显提高
2. challenge F1 小幅提高
3. 事件级命中率开始进入评估体系

第二点：V3 是当前最稳健的主版本

原因有三条：

1. 它在 test 和 challenge 上都取得了当前最好的综合平衡结果之一
2. 它的 precision 明显优于 V2 和 V4
3. 它的 anomaly_ratio 最低，最符合“误报控制优先”的目标

第三点：V4 不是失败版本，而是专项补充版本

1. 它在 test 集 F1 上略高于 V3
2. 它显著提升了 pause_hit_rate
3. 但 challenge F1 低于 V3
4. 它通过抬高 recall 和异常比例换取更强的事件覆盖

因此，V4 当前更适合被写成：

1. 一个有效的分类型优化尝试
2. 一个能补 pause 弱项的补充路线
3. 而不是最终统一主版本

5. 当前固定结论

后续实验中建议固定如下版本定位：

1. 主版本：V3
2. 补充版本：V4
3. 早期改进链路支撑：V1 -> V2

这样整理后的叙述逻辑最清晰：

1. V1 证明模型具备粗定位能力
2. V2 证明事件级评估和基础后处理有效
3. V3 解决了“故障邻域误报过多”的核心问题，因此成为主版本
4. V4 进一步说明可以沿着 fault-aware 方向修复 pause 弱项

6. 在最终报告中的推荐写法

建议把版本结论写成：

1. V3 是当前最适合部署或作为论文主结果展示的版本
2. V4 是有研究价值的专项修正版本
3. V4 没有否定 V3，反而补充说明了当前模型还存在故障类型不均衡的问题

7. 一句话总结

当前后续实验最应该展示的不是“哪个版本单方面全胜”，而是：在同一批实验数据上，模型性能沿着 V1 -> V2 -> V3 -> V4 逐步变得更可解释，而最终主版本之所以选择 V3，是因为它在误报控制和跨阶段稳定性上最平衡。
